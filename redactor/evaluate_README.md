# 評価指標の説明（TP/FP/FN が位置ベースである理由）

## なぜ「同じ要素」があるのに TP=0 になるか（test_doc_01 の例）

- **answer（正解）**: トークンが **6個** → セグメント数 **13**（リテラルとトークンが交互）
  - `<PERSON>`, `<LOCATION>`, `<PHONE_NUMBER>`, `<PHONE_NUMBER>`, `<EMAIL_ADDRESS>`, `<ID>`
- **redactor 出力**: トークンが **3個**のみ → セグメント数 **7**
  - 「ゆいと」「住所」「顧客ID」をマスクしていないため、それらが 1 つの長いリテラルになる
  - トークンは `<PHONE_NUMBER>`, `<PHONE_NUMBER>`, `<EMAIL_ADDRESS>` の 3 つだけ

評価は **「セグメントのインデックス位置」で 1:1 対応** を見ている。

- 正解の位置 1 → `<PERSON>`、実際の位置 1 → `<PHONE_NUMBER>` → **不一致**（FN + FP）
- 正解の位置 3 → `<LOCATION>`、実際の位置 3 → `<PHONE_NUMBER>` → **不一致**
- 正解の位置 5 → `<PHONE_NUMBER>`、実際の位置 5 → `<EMAIL_ADDRESS>` → **不一致**

このように、**マスク数が違うとセグメント列の長さ・並びが変わり、同じ種類のトークンでも「位置」がずれて全て不一致**になる。その結果、**TP=0, FP=3, FN=6** となる。

## 現在の評価方法の前提

- 正解と出力で **マスクする箇所の数・順序が同じ** ことを前提にした **厳密な位置一致** である。
- したがって「同じ要素（同じ種類のトークン）が含まれている」だけでは TP は増えず、**どこに何個トークンがあるか** が一致しないと TP は 0 のままになる。

## 評価方法は「正しい」か

- **設計どおりには正しい**：位置ベースで TP/FP/FN を数えている。
- **用途としては限定的**：マスク漏れ・過剰マスクがあると、セグメント構造が変わり、**意味的には正しいマスクでも TP にカウントされない**。

意味的な一致（トークン種類・出現回数）も見たい場合は、**トークン種類ベースの指標**を追加した。

### 種類ベース（位置不問）の評価

- 正解・出力それぞれで「どのエンティティ型が何個あるか」だけを集計する。
- 型 T ごとに: **TP += min(正解の個数, 出力の個数)**、**FN += 不足分**、**FP += 過剰分**。
- 位置は無視するので、マスク数が違っても「同じ型を同じ個数マスクした」分は TP になる。

例（test_doc_01）:
- 正解: PERSON=1, LOCATION=1, PHONE_NUMBER=2, EMAIL_ADDRESS=1, ID=1
- 出力: PHONE_NUMBER=2, EMAIL_ADDRESS=1（他は未マスク）
- → TP=3, FN=3, FP=0 → Precision 100%, Recall 50%, F1 66.67%

評価スクリプト実行時は **位置ベース**・**種類ベース**・**行ベース** の3つを表示・ファイル出力する。

### 行ベース（行ごとトークン種類）の評価

- 正解・出力を **行で分割** し、**同じ行番号の行どうし**でだけ比較する。
- 各行内で「どの型が何個あるか」を集計し、型ごとに TP/FN/FP を積算（種類ベースと同じ計算を行単位で実施）。
- 例: 正解の4行目に PERSON が1個、出力の4行目にトークンなし → その行では FN=1。正解の8行目と出力の8行目にそれぞれ PHONE_NUMBER が1個ずつ → その行では TP=1。
- **「どの行に何をマスクしたか」** を考慮するため、種類ベースより位置に近いが、行内の順序は問わない。
